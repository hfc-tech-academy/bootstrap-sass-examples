Bootstrap-sass on Rails

Forms

Forms are about collecting data from the user, and are a major part of any site. There are also some rules that you need to follow to keep your site accessible to visually impaired users using screen readers. Both rails and bootstrap have opinions on how to lay out forms and there are a couple of things that you need to do to make them work happily together.

There are three main bootstrap form styles. A basic form (using form with no class), inline form (<form class="form-inline">), and a horizontal form (<from class="form-horizontal">).

Inside of the form, each label/input pair should be in a form-group.

For form-inline and form-horizontal, each of the inputs should be tagged with the form-control class.

A basic form has the labels immediately above the inputs.

The inline form generally doesn't display labels (though it should have them), and the inputs are layed out horizontally (above $screen-sm-min or 768px) with placeholders used for form "labels". The labels in these from should have the sr-only class, which stands for screen reader only. The text will not be visible, but it will be available for screen readers.

A horizontal form has the labels to the left of the inputs for larger displays, and above the inputs for smaller displays. labels should have a control-label as well as a grid column class. The inputs should we wrapped in a div with a grid column class and have the form-control class. In a horizontal form the form-group acts as like a .row.

On our projects, the horizontal form is by far the most used.

To have a paragraph of text in the form itself, it should be in the form of <p class="form-control-static"> to make it line up correctly.

So, to sum up:

<form class="form-horizontal">
  <div class="form-group">
    <label for="email" class="control-label col-sm-2">Email</label>
    <div class="col-sm-10">
      <input type="text" class="form-control" id="email">
    </div>
  </div>

  <div class="form-group">
    <div class="col-sm-10 col-sm-offset-2">
      <p class="form-control-static">We hate spam too.</p>
    </div>
  </div>

  <div class="form-group">
    <div class="col-sm-offset-2 col-sm-10">
      <button type="submit" class="btn btn-default">Sign in</button>
    </div>
  </div>
</form>

.form-group acts like a .row, so when the screen gets smaller the label will pop up above the input type. Remember to wrap the input in a div - the widths of all input elements are 100% so if they look too big that's probably the reason.

Checkboxes and radio buttons

Checkboxes and radio buttons switch the order of the label and the control - visually the control is on the left side of the text. The way to do this is to wrap a div with the class of checkbox or radio around the label, and put the input inside of the label itself. You can also put grid classes on that outer div to make sure that it algins with the overall form grid.

<div class="form-group">
  <div class="checkbox col-sm-10 col-sm-offset-2">
    <label>
      <input type="checkbox" value="">
      Remember me?
    </label>
  </div>
</div>

Input groups

Sometimes you want to style the input forms with a little more information. An input group will let you put text on the front or the back of the form. For example, say you wanted to prompt someone for their twitter handle by having an @ sign at the beginning of what they type:

This is done by wrapping the input with another div with the class input-group. We can't use the existing class that we used for the grid layout; it needs to be nested:

<div class="form-group">
  <label for="twitter" class="control-label col-sm-2">Twitter</label>
  <div class="col-sm-10">
    <div class="input-group">
      <span class="input-group-addon">@</span>
      <input type="text" class="form-control" id="twitter">
    </div>
  </div>
</div>

Addons can be placed both before and after the input tag.

You can also place controls inside of an input group, including checkboxes, radio buttons.

If you want to place buttons inside, you need to use an additional <span class="input-group-btn">, best described in the button addon documentation.

Validation states

The main challenge when getting forms to work right with rails is the validation states. Bootstrap has two different things we can do: highlight the fields with the success, warning, or error, and optionally add icons within the text elements to show graphical feedback.

Validation states work by adding an additional class on the form-group element; the labels and inputs contained within will have the correct coloring. These classes are has-success, has-warning, and has-error. For example:

<form class="form-horizontal">
  <div class="form-group has-success">
    <label for="email" class="control-label col-sm-2">Email</label>
    <div class="col-sm-10">
      <input type="text" class="form-control" id="email">
    </div>
  </div>

  <div class="form-group has-warning">
    <label for="email" class="control-label col-sm-2">Email</label>
    <div class="col-sm-10">
      <input type="text" class="form-control" id="email">
    </div>
  </div>

  <div class="form-group has-error">
    <label for="twitter" class="control-label col-sm-2">Twitter</label>
    <div class="col-sm-10">
      <div class="input-group">
        <span class="input-group-addon">@</span>
        <input type="text" class="form-control" id="twitter">
      </div>
    </div>
  </div>

  <div class="form-group">
    <div class="col-sm-offset-2 col-sm-10">
      <button type="submit" class="btn btn-default">Sign in</button>
    </div>
  </div>
</form>

If we want to add some icons on the field, we need to add an additional class of has-feedback and then the icon itself near input:

<form class="form-horizontal">
  <div class="form-group has-success has-feedback">
    <label for="email" class="control-label col-sm-2">Email</label>
    <div class="col-sm-10">
      <input type="text" class="form-control" id="email">
      <span class="glyphicon glyphicon-ok form-control-feedback"></span>
    </div>
  </div>

  <div class="form-group has-warning has-feedback">
    <label for="email" class="control-label col-sm-2">Email</label>
    <div class="col-sm-10">
      <input type="text" class="form-control" id="email">
      <span class="glyphicon glyphicon-warning-sign form-control-feedback"></span>
    </div>
  </div>

  <div class="form-group has-error has-feedback">
    <label for="twitter" class="control-label col-sm-2">Twitter</label>
    <div class="col-sm-10">
      <div class="input-group">
        <span class="input-group-addon">@</span>
        <input type="text" class="form-control" id="twitter">
      </div>
      <span class="glyphicon glyphicon-remove form-control-feedback"></span>
    </div>
  </div>

  <div class="form-group">
    <div class="col-sm-offset-2 col-sm-10">
      <button type="submit" class="btn btn-default">Sign in</button>
    </div>
  </div>
</form>

Notice that we had to add the icon span after the input-group.

Validation in rails

If you use the form_for helpers in rails to generate your form, when there is a validation error it automatically generates an extra <div class="field_with_errors"> around your inputs to mark it as an error. This div is no longer needed and isn't where bootstrap needs to have the class.

We can remove this behavior by editing config/application.rb and adding this config option:

config.action_view.field_error_proc = Proc.new { |html_tag, instance| html_tag }

You'll need to restart rails for this to take effect.

Lets now add some helper methods inside of application_helper.rb

  def has_error?( model_class, attribute )
    model_class.errors[attribute].size > 0
  end

  def error_for( model_class, attribute, err_class )
    return err_class if has_error?
    nil
  end

Now we can implement the form and have it add the errors in the correct place:

<%= form_for @user, :html => { :class => "form-horizontal" } do |f| %>
  <!-- Error Messages -->
  <% if @user.errors.any? %>
    <div class="alert alert-danger alert-dismissable">
      <button aria-hidden="true" class="close" data-dismiss="alert" type="button">&times;</button>
      <h4>
        <%= "#{pluralize(@user.errors.count, "error")} prohibited this user from being saved:" %>
      </h4>
      <ul>
        <% @user.errors.full_messages.each do |msg| %>
          <li>
            <%= msg %>
          </li>
        <% end %>
      </ul>
    </div>
  <% end %>

  <div class="form-group <%= has_error?( @user, :email1 ) ? "" : "has-success" %>">
    <%= f.label :email1, :class => 'col-sm-2 control-label' %>
    <div class="col-sm-10">
      <%= f.text_field :email1, :class => 'form-control' %>
    </div>
  </div>

  <div class="form-group <%= error_for( @user, :email2, "has-warning" ) %>">
    <%= f.label :email2, :class => 'col-sm-2 control-label' %>
    <div class="col-sm-10">
      <%= f.text_field :email2, :class => 'form-control' %>
    </div>
  </div>

  <div class="form-group <%= error_for( @user, :twitter, "has-error" ) %>">
    <%= f.label :twitter, :class => 'col-sm-2 control-label' %>
    <div class="col-sm-10">
      <div class="input-group">
        <span class="input-group-addon">@</span>
        <%= f.text_field :twitter, :class => 'form-control' %>
      </div>
    </div>
  </div>


  <div class="form-group">
    <div class="col-sm-offset-2 col-sm-10">
      <%= f.submit :class => 'btn btn-primary' %>
    </div>
  </div>
<% end %>

This does a few things. First we are displaying all of the error messages if they exist in a list. The we are using the error_for helpers to check to see if there are any messages for a specific attribute. (For :email1 we are checking for the absense of errors to display the has-success, which doesn't make that much sense but it keeps the example consistent.) If an error is found, the helper emits the third parameter, so you can choose if it's a has-error or has-warning.

To add the icons, you need to add an additional <%= error_for( @user, :twitter, "has-feedback" ) %> block inside of the form-group class, and then add the check after each of the inputs like so:

<% if has_error?( @user, :twitter ) %>
  <span class="glyphicon glyphicon-remove form-control-feedback"></span>
<% end >
