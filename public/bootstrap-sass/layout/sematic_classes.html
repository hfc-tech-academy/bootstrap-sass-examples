Bootstrap-sass on Rails

Semantic Classes

WikiPedia defines semantic HTML as

Semantic HTML is the use of HTML markup to reinforce the semantics, or meaning, of the information in webpages rather than merely to define its presentation or look. Semantic HTML is processed by regular web browsers as well as by many other user agents. CSS is used to suggest its presentation to human users."

For the record, we always focus on human users.

This suggests that it is better to mark up your HTML using classes whose names reflect what they are for. For example, say we wanted to lay out a basic blog site with 3 columns. A left with navigation elements, a centered column with the blog article, and then a right column for ads or suggestions of things that the user may also like.

(I'm adding the bordered class here so you will be able to see the layout.)

.bordered { border: 1px solid gray; }

<div class="container">
  <header>
    <nav class="bordered">
      Navbar
    </nav>
  </header>

  <section class="content">
    <div class="sidebar bordered">
      left gutter
    </div>

    <article class="bordered">
      main content area
    </article>

    <div class="advertisement bordered">
      right gutter
    </div>
  </section>

  <footer class="bordered">
    Footer
  </footer>
</div>

Just by looking at this code we have a sense of what it's trying to accomplish. The HTML tags and CSS classes are a reflection of the intent of what we are trying to achieve, and there's not a lot of mysteriousness of what's in each section.

How does this fit in with bootstrap? If we wanted to layout something similar with bootstrap, are we forced to do something like the following?

<div class="container">
  <div class="navbar bordered">
    Navbar
  </div>

  <div class="row">
    <div class="col-md-2 bordered">
      left gutter
    </div>

    <div class="col-md-8 bordered">
      main content area
    </div>

    <div class="col-md-2 bordered">
      right gutter
    </div>
  </div>

  <div class="footer bordered">
    Footer
  </div>
<div>

Surely not! Bootstrap is a CSS Framework and we don't need to limit outselves to the built in classes. In fact, there's a bunch of things that we can do using the SASS mixins directly that aren't possible by using the built in classes. Lets take a look at how we'd format the above.

@import 'bootstrap-sprockets';
@import 'bootstrap';

.bordered {
  border: 1px solid gray;
}

nav {
  @extend .navbar;
}

section.content {
  @extend .row;

  .sidebar {
    @extend .col-md-2;
  }

  article {
    @extend .col-md-8;
  }

  .advertisement {
    @extend .col-md-2;
  }
}

footer {
  @extend .footer;
}

If we load the pages, we can see that semantic HTML is rendered the same as the "using the built in classes" way of doing things, while retaining its readability.

@extend is a SASS function. @extend .navbar; takes all of the properties defined in .navbar and applies them to, in this example, the nav element. section.content is really just another row, so we include all of those definations in there as well and don't need to add a wrapping class.

Similarily, we are using the grid definations for the .sidebar, article, and .advertisement divs. The definations will get includes, so it will be just as if we did <div class="col-sm-2"> and <div class="col-md-8">.

Dealing with Responsive Classes

Let's expand this example. Lets say for small screens, we want the right column to disappear, and the left and center column to be spaced 3-9. Here's the straight bootstrap way:

<div class="container">
  <div class="navbar bordered">
    Navbar
  </div>

  <div class="row">
    <div class="col-md-2 bordered col-sm-3">
      left gutter
    </div>

    <div class="col-md-8 bordered col-sm-9">
      main content area
    </div>

    <div class="col-md-2 bordered hidden-sm hidden-xs">
      right gutter
    </div>
  </div>

  <div class="footer bordered">
    Footer
  </div>
<div>

We've added col-sm-3 and col-sm-9 to the left and center columns, and hidden-sm and hidden-xs on the right gutter.

Part of the push for sematic HTML is that there's something gross about this: why are we changing the HTML to change the layout? Shouldn't we be changing CSS instead?

When we mark up our HTML purely based upon our intent, the layout information is seperated from the markup, so in theory it can be changed independantly. There are times which a little tweaking is needed of the HTML, but in our case all we need to do is change the SASS. Lets adapt our SASS to for the semantic version:

section.content {
  @extend .row;

  .sidebar {
    @extend .col-sm-3;
    @extend .col-md-2;
  }

  article {
    @extend .col-sm-9;
    @extend .col-md-8;
  }

  .advertisement {
    @extend .hidden-sm;
    @extend .hidden-xs;
    @extend .col-md-2;
  }
}

A few things to point out: while the order of the CSS classes don't matter in the HTML, the order of the @extends do matter. Because of the cascading way that CSS and they way that the media queries are implemented in bootstrap you should always list the smaller column types before the larger ones. We've also had to @extend both the hidden-xs and the hidden-sm classes to hide the right column when the design is small. The order in this case doesn't matter because the media selectors are specific and don't fall through.
